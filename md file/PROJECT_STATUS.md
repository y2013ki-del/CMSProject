> 읽는 시점: 프로젝트 전체 방향, 현재 구현 범위, 우선순위, 운영 리스크를 빠르게 파악할 때
> 관련 문서: `README.md`, `md file/DOC_INDEX.md`, `md file/UI_GUIDE.md`, `md file/API_REFERENCE.md`

## 현재 프로젝트 방향

현재 프로젝트는 `MagicInfo 대체형 온프레미스 CMS`를 목표로 하되, 구현 우선순위는 아래 4축에 맞춰져 있습니다.

1. 콘텐츠를 단순 파일이 아니라 `배치형 콘텐츠`로 설계하고 재편집 가능하게 유지
2. 스케줄을 직접 기기에 넣는 대신 `채널`을 기준으로 규칙 배포
3. 관리자 UI에서 운영자가 파일, 웹, 날씨, 레이아웃을 한 흐름 안에서 다룰 수 있게 통합
4. 삼성 QMC 장비 제어(MDC)와 플레이어 스케줄 반영을 한 서버에서 처리

즉 현재 핵심 플로우는 아래와 같습니다.

```text
콘텐츠 등록
  └── 파일 / 웹 / 날씨 / 배치형 콘텐츠
        ↓
스케줄 편집
  └── 콘텐츠 리스트와 재생 순서 구성
        ↓
채널 편집
  └── 기본 / 예약 / 요일 규칙으로 스케줄 배정
        ↓
기기 연결
  └── 디스플레이에 채널 연결
        ↓
플레이어 수신
  └── /client 가 WebSocket으로 채널 스케줄 수신
```

## 현재 구현 상태

### 1. 관리자 UI

- `기기 모니터링`
  - 등록/미등록 기기 분리
  - 기기 상태, 채널 연결, 재생 프로필 관리
- `기기 제어`
  - 개별/선택 기기 일괄 제어
  - 전원, 입력, 볼륨, 플레이어 재로드
- `콘텐츠`
  - 콘텐츠 그룹
  - 파일 업로드
  - 웹 콘텐츠 등록
  - 날씨 콘텐츠 추가
  - 콘텐츠 디자이너(가로/세로, 분할, 영역 추가, 비율 고정, 화면 맞춤)
  - 기존 콘텐츠 재편집
- `스케줄`
  - 스케줄 그룹
  - 스케줄 생성/수정/삭제
  - 콘텐츠 추가/정렬
- `채널`
  - 채널 생성/삭제
  - 기본/예약/요일 규칙
  - 우선순위
  - 색상 지정
  - 달력 보기
- `로그`
  - 카테고리 필터
  - 검색
  - 50줄 단위 페이지 이동
  - 기간 지정 다운로드

### 2. 콘텐츠 모델

현재 콘텐츠는 3가지가 아닙니다. 실제로는 아래처럼 나뉩니다.

- `일반 미디어`
  - 이미지 / 영상 원본 파일
- `웹 콘텐츠`
  - 외부 URL 또는 내부 URL
- `날씨 콘텐츠`
  - 내부 `weather-content.html` 기반 웹 콘텐츠
- `디자인 콘텐츠`
  - 여러 소스를 배치한 레이아웃형 콘텐츠
- `디자인 메타가 붙은 일반 미디어`
  - 일반 파일이지만 디자이너 수정 결과가 붙어 실제 재생 시 래퍼로 동작

### 3. 플레이어/송출

- `/client` 플레이어가 WebSocket `/ws/player` 로 채널 스케줄 수신
- 스케줄은 채널 기준으로 계산 후 플레이어에 전달
- 일반 파일이더라도 디자인 메타가 있으면 실제 송출 시 내부 `designer-content.html` 경로로 변환
- 콘텐츠 수정 후 운영자가 선택하면 `즉시 반영`으로 채널 재배포 가능

### 4. 제어/운영

- MDC TCP 1515 기반 전원/입력/볼륨 제어
- WoL 송신 지원
- 온라인 플레이어 상태 추적
- 구조화 로그 저장
- 요청자 IP(`actorIp`) 기반 작업자 추적

### 5. CCTV

- CCTV 코드는 아직 프로젝트 안에 존재
- 하지만 기본은 비활성
- `ENABLE_CCTV=1` 일 때만 관련 API와 경로를 실제 사용

즉 현재 프로젝트의 주 운영 방향은 CCTV가 아니라 `CMS 본체`입니다.

## 최신 설계 결정 사항

### 콘텐츠 디자이너

- 콘텐츠 등록은 단순 업로드가 아니라 디자이너 중심
- 업로드 후 기존 배치 수정 가능해야 함
- 날씨/웹 콘텐츠도 디자이너 안에서 같은 방식으로 다룸
- 디자이너 내부 숨김 원본 파일은 UI 노출보다 재편집 가능성을 우선

### 채널 우선 구조

- 스케줄은 재생 목록
- 채널은 운영 규칙
- 기기는 채널만 바라봄

즉 운영자 기준 개념을 아래처럼 분리합니다.

```text
콘텐츠 = 재생 소재
스케줄 = 소재 묶음
채널 = 실제 송출 규칙
기기 = 채널 수신 대상
```

### 로그 정책

- 로그는 `data/cms.log`
- 30일 기준 로테이션
- 변경성 관리자 작업에는 `actorIp` 기록
- 로그 UI는 최근 로그 조회용
- 기간 다운로드는 사후 분석용

## 현재 남아 있는 핵심 과제

### 우선순위 높음

1. 문서와 운영 흐름을 계속 현재 UI 기준으로 유지
2. 콘텐츠 디자이너 저장/수정/삭제 예외 케이스 추가 검증
3. 플레이어 재생 안정성 검증
4. 채널 규칙과 달력 표현 정합성 추가 점검

### 우선순위 중간

1. 로그 화면의 UX 추가 개선
2. 관리자 작업 이력에 `actorId` 추가 검토
3. 데이터 이전/배포 자동화 정리
4. 일반 파일과 디자인 콘텐츠의 표시 정책 명확화

### 우선순위 낮음

1. CCTV 재활성화 여부 재판단
2. 대규모 라이브/멀티캐스트 구조 별도 분리
3. AVPlay/NTP 정밀 동기화 확장

## 운영상 주의사항

### 1. 배포 폴더 교체 시 코드만 복사하면 안 됨

운영 상태를 유지하려면 아래가 같이 이동해야 합니다.

- `data/schedules.json`
- `data/channels.json`
- `data/displays.json`
- `data/media-groups.json`
- `data/web-contents.json`
- `data/media-meta.json`
- `media/library/`

### 2. 콘텐츠 수정 반영 시점

- 일반 수정만 하면 다음 재생부터 반영될 수 있음
- 즉시 반영이 필요하면 관리자 팝업에서 재배포 선택 필요

### 3. CCTV는 기본 비활성

- 문서나 운영 인수인계 시 CCTV가 항상 켜져 있다고 가정하면 안 됨
- 실제 사용 여부는 `ENABLE_CCTV` 환경에 따라 달라짐
